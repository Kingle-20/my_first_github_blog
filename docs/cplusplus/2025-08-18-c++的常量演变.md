## 一、前言

C++ 作为一门编译型语言，其显著优势之一在于提供了强大的编译期计算能力。开发者希望可以利用语言级别的特性，在编译阶段向编译器传递语义信息，从而实现运行时性能的优化和潜在错误的提前发现。

在程序设计中，单一职责原则指出：一个类应当只负责一项职责。类比到函数层面，一个函数应专注于完成一个明确的功能。这种职责单一性不仅提升了代码的可读性和可维护性，也为编译器优化创造了有利条件。当函数逻辑清晰、职责明确时，编译器更容易分析变量的使用模式和数据流，进而实施更有效的优化策略。

其中，一个关键的优化前提就是不变性：如果某个值在特定作用域内不会发生变化，编译器便可基于这一假设进行诸如常量折叠、公共子表达式消除、循环不变量外提等优化。而 C/C++ 作为能够直接操作底层硬件和内存的系统级编程语言，尤其强调运行效率。因此，开发者应主动利用“不变性”这一语义信息，通过显式声明常量、使用 const 限定符等方式，帮助编译器做出更优的代码生成决策。

正因如此，C/C++ 引入了常量的核心概念——它不仅是程序正确性的保障，更是性能优化的重要工具。通过合理使用常量，开发者不仅能表达设计意图，还能显著提升程序的执行效率与可预测性。

## 二、运行期不变--const

c++作为c子集，自然继承了c语言的大部分特性。在编写c语言时，如何向编译器表达出不变性呢？c标准为了表达变量运行期不变性这个概念，提出了`const`关键字。

C语言中没有类的源概念，因此它仅仅只是用于表示一个变量在变量作用域内不会发生变化。而c++对其进行了拓展，它可以用于成员变量。

### 2.1 修饰局部变量

在整个作用域期间都是不会变换的

### 2.2 修饰全局变量

在整个程序的运行期间都是不会变换的。在c++17以后，const变量默认由于其整个运行期间都不会发生变量，默认具有内联性质。因此如果在头文件中直接定义一个const类型的全局变量，是可以的。

### 2.3 修饰类成员函数

表示该成员函数不会修改类的数据成员。c++认为const修饰符是作为函数重载标签中的一部分的。这也就意味着该成员函数如果要调用其他成员函数，也必须要有const修饰符修饰。

那么为什么要加上修饰成员函数呢？

很多时候，可能会存在使用const来修饰一个自定义类型，并且为了保证类的封装特性，就需要使用这个自定义类型的成员函数。如何保证这个成员函数是能够不修改对象呢？那就是使用const修饰。

## 二、编译期不变半成品--constexpr

前面所提到的运行期不变并不能进一步优化代码。很多时候代码中是可以在编译期就可以计算出对应的数值的。比如说

```c++
int sum = 0;
for(int i = 0; i < 10; i ++) {
    sum += i;
}

```

很多时候为了程序方便修改，i的最大值可能随着场景或者是需求的不同会发生不一样的变化。这个时候可能是10，也可能是100。那么我们很希望它是在编译期就计算完毕的。因此c++11就提出了constexpr这个关键字。

同样它可以修饰变量，也可以修饰成员函数。但是为了实现编译期计算，并且方便封装，c++提出constexpr可以修饰普通函数，无论是否为模板普通函数。

### 2.1 修饰基础类型变量

constexpr可以修饰基础类型变量，它的功能作用是帮助程序员不再使用宏定义，并且将这个基础类型字面量赋予变量名称，以便于写出可维护的代码。

按照作用域和位置不同分为两种：

1. 全局域和局部域  
    `constexpr int xx = 10;`

2. 类内部  
    `constexpr static int xx = 10;`  
    需要使用`static`修饰。

### 2.2 修饰自定义类型

当constexpr修饰自定义类型，其含义是：该对象必须在编译期完成构造，且其值在程序运行前就已确定。

常见如下几种：

1. 聚合类
2. 如果类需要自定义构造函数（例如进行参数验证、计算默认值等），则该构造函数必须显式声明为 constexpr，才能用于构造 constexpr 对象。

对于自定义类型，被constexpr修饰的构造函数并不意味着一定只能构造constexpr的对象，它可以构造一般的对象。

### 2.3 修饰普通函数

最开始c++认为constexpr对函数修饰的含义是，尽可能使用该函数在编译期进行求值。如果这个函数的返回值被一个constexpr修饰的变量接收时，c++会强制要求这个函数求值必须要能够在编译期求值，否则报错。

因此，一个函数被constexpr修饰时，并不一定要求在编译期完成。可能当时是为了提高一个函数的可用程度吧。

### 2.4 修饰if的条件编译

有的时候，一个条件判断是编译期已知的，但是又想要写出一般化的代码，以方便写出可以维护的代码。

为了优化这种场景，C++17 引入 if constexpr 语句，允许在编译期对常量表达式进行条件判断。与普通 if 不同，if constexpr 的分支会在编译时根据条件值进行裁剪：不满足的分支不会被实例化，仅保留有效代码。这使得开发者可以编写泛型、通用的代码逻辑，同时避免运行时开销。例如，在模板编程中可根据类型特性选择不同实现路径，既保证了代码的可维护性和复用性，又实现了零成本抽象，是现代 C++ 实现静态多态和编译期优化的重要工具。


## 三、编译期强制求值--consteval

constexpr只能保证尽量求值，如果让编译期保证一定求值呢？

c++20为了提供这种场景，便提出了consteval这个关键字，它专门用于一般函数，表达一定在编译期求值的语义，如果不能，就要报错处理。

consteval专门用于非成员函数中，要求该函数一定在编译期能够求值。
