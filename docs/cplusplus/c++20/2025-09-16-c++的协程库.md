## 一、前言

在现代软件系统中，尤其是高并发、异步 I/O、事件驱动等场景下，传统的同步编程模型面临严峻挑战。硬件发展出现了如何特征：

1. cpu单核频率提升速度越来越受限，cpu厂商越来越强调多核。

2. IO操作的速度相较于单核cpu本身的速度慢很多。

3. 高并发的需求越来越大。

第三条是场景需求，前两条是硬件限制。因此为了提高并发量，异步IO的设计要求变得越来越强烈。

### 1.1 传统异步回调解决方案

在现代高性能系统中，异步 I/O 是实现高并发、低延迟的关键技术。传统的异步编程模型通常依赖 线程池 + 回调函数（Callback） 的方式来处理耗时操作（如文件读写、网络请求、数据库查询等）。

核心思想是主线程不直接执行耗时任务；而是将任务封装为函数对象（回调），提交给线程池；线程池中的工作线程执行任务；任务完成后，通过预设的回调通知主线程或继续后续逻辑；主线程通过某种机制（如共享状态、future、事件循环）获取执行结果。

当业务逻辑变得复杂，尤其是涉及多级嵌套、分支选择、条件执行时，回调层层嵌套，导致：

1. 代码难以阅读和维护；
2. 错误处理困难；
3. 局部变量生命周期管理复杂；
4. 调试困难；
5. 无法使用 return、try/catch 等自然控制流。

如下面模拟一部数据路查询：

```c++
#include <iostream>
#include <string>
#include <functional>
#include <thread>
#include <future>
#include <memory>

// 模拟线程池
class ThreadPool {
public:
    template<typename F>
    void submit(F&& func) {
        std::thread t(std::forward<F>(func));
        t.detach(); // 简化模型，不管理线程生命周期
    }
};

// 全局线程池实例
ThreadPool g_thread_pool;

// 模拟异步数据库查询
void async_query_db(std::string sql, std::function<void(std::string)> callback) {
    g_thread_pool.submit([sql, callback = std::move(callback)]() {
        std::this_thread::sleep_for(std::chrono::seconds(1)); // 模拟 I/O 延迟
        std::string result = "data_from_db: " + sql;
        std::cout << "[DB] 查询完成: " << result << std::endl;
        callback(result); // 回调通知
    });
}

// 模拟异步缓存查询
void async_get_cache(std::string key, std::function<void(std::string)> callback) {
    g_thread_pool.submit([key, callback = std::move(callback)]() {
        std::this_thread::sleep_for(std::chrono::milliseconds(300));
        std::string result = "cache_hit: " + key;
        std::cout << "[Cache] 获取成功: " << result << std::endl;
        callback(result);
    });
}

// 模拟异步发送响应
void async_send_response(std::string response, std::function<void()> on_done) {
    g_thread_pool.submit([response, on_done = std::move(on_done)]() {
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
        std::cout << "[Network] 响应已发送: " << response << std::endl;
        if (on_done) on_done();
    });
}

// 主要业务逻辑：处理用户请求
void handle_user_request(std::string user_id) {
    std::cout << "开始处理用户请求: " << user_id << std::endl;

    // 第一步：先查缓存
    async_get_cache("user:" + user_id, [user_id](std::string cache_result) {
        if (cache_result.find("hit") != std::string::npos) {
            // 缓存命中：直接发送响应
            async_send_response("Response: " + cache_result, [] {
                std::cout << "请求处理完成（缓存路径）" << std::endl;
            });
        } else {
            // 缓存未命中：查数据库
            async_query_db("SELECT * FROM users WHERE id = " + user_id, 
            [user_id](std::string db_result) {
                // 数据库查完，再写入缓存（模拟）
                async_get_cache("user:" + user_id + "_write", [db_result](std::string) {
                    // 最后发送响应
                    async_send_response("Response: " + db_result, [] {
                        std::cout << "请求处理完成（数据库路径）" << std::endl;
                    });
                });
            });
        }
    });
}
```

它的问题是什么呢？

上述代码虽然功能正确，但存在以下问题：

1. 嵌套层级深：三层嵌套回调，逻辑分散；
2. 控制流不清晰：if (cache hit) 和 else 分支在不同回调中；
3. 错误处理困难：没有异常传播机制，每个回调都要单独处理错误；
4. 变量捕获复杂：user_id 需要层层传递；
5. 调试困难：堆栈断裂，无法像同步代码一样单步调试。

那么能否实现<b>“高并发 + 顺序式编程体验”</b>抽象概念呢？

那么核心诉求为：

+ ✅ 可暂停与恢复：函数执行到某点可暂停，稍后从该点继续；
+ ✅ 轻量级：创建成本远低于线程；
+ ✅ 顺序式语法：代码看起来像同步，实则异步执行；
+ ✅ 资源高效：不阻塞线程，支持数万级并发任务。


## 二、无栈协程

c++提出了无栈协程概念。至于有栈无栈的优缺点，本文不涉及，毕竟无栈协程都进标准了，说明其还是有一定的优越性。

### 2.1 初识协程

为了实现无栈协程，C++20 引入了五个关键抽象部分：

1. 协程返回对象
2. 协程桢
3. promise_type 对象
4. promise_handle 对象
5. awaiter 对象

设计如此多模块的主要原因在于，协程本质上是在模拟一种用户态的线程。

这涉及协程使用者如何进行异步操作和获取结果这两个核心关注点；协程实现者如何构造出形式上类似同步代码的异步逻辑；以及编译器如何将协程转换为对应的二进制代码。

在前面提到，一种常见的异步解决方案是：

+ 主线程将 I/O 任务提交给线程池；
+ 随后主线程继续执行其他操作；
+ 之后主线程通过主动查询，或由线程池以某种方式通知，得知任务完成；
+ 最后主线程获取执行结果。

根据上面的过程，来明确协程使用者，协程实现者以及编译器各自关心的内容。

对于协程使用者而言有如下问题：

1. 怎么提交异步任务？

2. 怎么获取协程的执行结果？

对于协程实现者而言有如下问题：

1. 在将协程使用者提交的任务提交到线程池之后，如何暂停当前“仿同步”的代码，然后返回，从而交还线程的控制权。

2. 当线程池完成任务之后，如何恢复任务，然后执行后续的任务。

而对于编译器而言：

1. 提供哪些功能，能够既满足协程使用者的需求，又能够满足协程实现者的需求。

对于c++编译器而言，首先满足协程使用者的需求：

+ 协程使用者通过调用一种特殊的函数，从而提交异步任务。

+ 协程使用者接受这种特殊的函数的返回值，调用特定的方法来获取异步任务的结果。

然后对于协程实现者而言，编译器提供了三个运算符来完成同步代码异步化：

+ co_await

+ co_return

+ co_yield

### 2.2 示例代码

由于协程各个部分联系比较紧密，难以通过分开拆解各自内容，先看一段代码，通过整体来感受下协程是如何执行的，然后再看各个部分的意义以及工作方式。

```c++
#include <coroutine>
#include <iostream>

#include <coroutine>
#include <iostream>

// 自定义 awaiter：第一次不挂起，第二次挂起
struct suspend_later_awaiter {
    bool should_suspend = true; // 第一次 ready=true，之后变为 false 并开始挂起

    bool await_ready() {
        if (should_suspend) {
            std::cout << "awaiter: 第一次 await_ready() 返回 true，不挂起\n";
            should_suspend = false;
            return true; // 不挂起
        } else {
            std::cout << "awaiter: 第二次 await_ready() 返回 false，准备挂起\n";
            return false; // 挂起
        }
    }

    void await_suspend(std::coroutine_handle<> handle) {
        std::cout << "awaiter: 协程被挂起，稍后恢复...\n";
        // 立即恢复（模拟异步操作完成）
        handle.resume();
    }

    void await_resume() {
        std::cout << "awaiter: 恢复执行\n";
    }
};

// 1. 定义返回类型 task<int>
struct task {
    struct promise_type {
        int value;
        std::suspend_always initial_suspend() { return {}; }
        std::suspend_always final_suspend() noexcept { return {}; }
        task get_return_object() { return task{this}; }
        void return_value(int v) { value = v; }
        void unhandled_exception() {}
    };

    promise_type* p;
    std::coroutine_handle<promise_type> handle() const {
        return std::coroutine_handle<promise_type>::from_promise(*p);
    }

    int result() {
        handle().resume(); // 恢复协程直到完成
        return p->value;
    }

    ~task() {
        if (handle()) handle().destroy();
    }
};

// 2. 定义协程函数（使用自定义 awaiter）
task compute_value() {
    std::cout << "协程开始执行\n";
    suspend_later_awaiter awaiter;
    // 第一次 co_await：不挂起
    co_await awaiter;
    std::cout << "第一次 co_await 完成\n";

    // 第二次 co_await：会挂起再恢复
    co_await awaiter;
    std::cout << "第二次 co_await 完成\n";

    co_return 42;
}

// 3. 主函数调用
int main() {
    std::cout << "创建协程\n";
    task t = compute_value(); // 创建协程，initial_suspend 挂起

    std::cout << "获取结果（驱动协程）\n";
    int result = t.result(); // 恢复协程运行

    std::cout << "最终结果: " << result << std::endl;
    return 0;
}

```

其执行结果如下：

```
创建协程
获取结果（驱动协程）
协程开始执行
awaiter: 第一次 await_ready() 返回 true，不挂起
awaiter: 恢复执行
第一次 co_await 完成
awaiter: 第二次 await_ready() 返回 false，准备挂起
awaiter: 协程被挂起，稍后恢复...
awaiter: 恢复执行
第二次 co_await 完成
最终结果: 42
```

表面上看，输出顺序略显混乱，但理解协程的关键在于分清关注层次。对于未知代码，应首先从使用者的角度了解其使用方式。

从使用者视角来看，真正相关的行为只有以下三行输出：

```
创建协程
获取结果（驱动协程）

最终结果: 42
```

在 main 函数中，首先调用 compute_value() —— 这是一个协程函数。调用后立即返回一个协程返回对象 task t，表示异步任务已提交，但并未阻塞当前线程。随后，通过调用 t.result() 驱动协程执行并等待结果。

这一过程符合典型的异步操作流程：

1. 提交任务（compute_value() 调用）；
2. 后续通过接口查询状态或获取结果（t.result()）；
3. 最终取得执行结果。

其中，协程使用者仅需关心协程返回对象 task 及其 result() 方法。只要最终能正确获得结果，且行为表现如同串行执行，就满足了使用者的预期。协程内部的挂起、恢复等细节，对使用者透明。

而协程实现者便要关心如何在标准或者换句话说，在编译器提供的功能完成仿佛同步的异步代码。

1. 协程返回对象如何实现
2. 协程桢大致结构
3. promise_type 对象
4. promise_handle 对象
5. awaiter 对象

### 2.3 模块构成

现在讲一下前面提到的协程的五个关键抽象，先讲讲它们各自的职责：

1. 协程返回对象：协程函数的返回值类型，供使用者使用，用于获取协程的执行结果或控制其行为。
2. 协程帧（Coroutine Frame）：由编译器在堆上分配的内存块，用于保存协程的参数、局部变量、暂停状态以及恢复执行所需的所有上下文信息。
3. promise_type 对象：定义在协程返回对象中的嵌套类型，是协程内部逻辑的“控制器”，负责管理协程的行为，如初始化、暂停、返回值处理和异常传播。
4. coroutine_handle 对象：一种不持有资源的轻量级句柄，用于对协程帧进行低层操作，如恢复执行（resume）或销毁帧（destroy），是协程外部控制流程的主要工具。
5. awaiter 对象：与 co_await 表达式相关联的对象，用于定义等待操作的行为，包括是否阻塞、如何挂起协程以及恢复后的后续动作。

这些组件共同支撑了无栈协程的运行机制：

1. 协程函数作为入口，调用时会创建协程帧，并在其中构造 promise_type 对象来协调内部逻辑。

2. 协程返回对象由 promise_type 生成，供外部使用。

3. 通过 co_await，协程可在异步操作未完成时挂起自身，将控制权交还调用者；当条件满足后，通过 coroutine_handle 恢复执行。

4. 整个过程由编译器在背后生成状态机代码来实现，使得异步逻辑能以同步形式书写。

## 三、协程组件——promise_type

当一个函数中出现了 co_await、co_return 等协程关键字，这意味着它是一个协程。编译器会为该函数生成状态机代码，并在运行时于堆上分配协程帧，用于保存参数、局部变量和当前执行状态。

协程的创建和控制依赖 promise_type。它是协程返回类型（如 task）中定义的嵌套类型，由编译器在协程启动时自动实例化。通过 promise_type 提供的方法，如 initial_suspend、final_suspend、get_return_object 等，协程的初始化、挂起行为和返回对象的生成得以定制。

因此，promise_type 起到了控制协程行为的核心作用，相当于协程的“控制芯片”，决定了协程如何被创建、暂停和销毁。

关键方法如下：

| 方法 | 作用 |
|------|------|
| `get_return_object()` | 创建并返回协程返回对象 |
| `initial_suspend()` | 初始挂起点：决定是否立即运行 |
| `final_suspend()` | 最终挂起点：决定是否自动销毁 |
| `return_value(T)` / `return_void()` | 处理 `co_return` |
| `unhandled_exception()` | 捕获未处理异常 |

### 3.1 promise_type控制器作用

promise_type 通过其特定成员函数在协程的关键节点被编译器自动调用，从而实现对协程执行流程的控制。具体过程如下：

当一个函数被标记为协程后，编译器会根据其返回类型中的嵌套类型 promise_type，在协程启动前于协程帧中创建一个 promise_type 实例。

创建完成后，立即调用该实例的 get_return_object() 成员函数，生成协程的返回对象（如 task）。这个对象返回给调用者，用于后续获取结果或控制协程。

接着调用 initial_suspend()，决定协程是否在开始时挂起：

+ 返回 std::suspend_never：协程立即执行函数体；
+ 返回 std::suspend_always：协程挂起，控制权立即返回给调用者。

在协程函数中执行 co_return value; 时，编译器会调用 promise_type 的 return_value(value) 成员函数，将返回值保存到 promise 对象中，以便后续读取。

co_return 完成后，调用 final_suspend()，决定协程结束后是否挂起：

返回 std::suspend_never：协程继续执行，协程帧被销毁，promise_type 对象也随之销毁；
返回 std::suspend_always：协程挂起，协程帧和 promise_type 对象保持有效，直到外部显式恢复或销毁。
通过这些成员函数，promise_type 实现了对协程创建、启动、值传递和销毁全过程的精细控制。

## 四、协程返回对象（Return Object）：协程的“代理”

协程返回对象是协程帧的访问器或控制器，其内部保有获取协程帧的控制能力。具体而言，协程返回对象通过调用协程内部的嵌套类（promise_type）来间接构造，以此在协程和协程返回对象之间建立关系。值得注意的是，它并不像常规的函数那样，是编译器直接调用返回值的构造函数，然后完成构造。这是因为promise_type并不要求提供供协程外部操作的成员函数，因此协程返回对象的设计重点就在于实现对整个协程的获取与控制。

### 4.1 构造过程

通过协程内部的promise_type嵌套类构造，具体构造过程为：

+ 协程开始执行时，编译器会创建promise_type对象
+ 调用promise_type::get_return_object()获取协程返回对象
+ 该对象随后被返回给协程的调用者

### 4.2 职责

1. 协程生命周期管理
    + 持有coroutine_handle：保存协程的句柄，用于恢复、销毁等操作
    + 控制执行权：提供resume()/`operator ()`方法恢复协程的执行
    + 资源管理：确保协程资源在适当时候被释放（通过RAII机制）
    + 状态跟踪：维护协程的当前执行状态（挂起、完成等）

2. 与promise_type的协作
    + 结果传递：通过promise_type设置协程的返回值或异常
    + 异常处理：捕获并传播协程中抛出的异常
    + 协程间通信：作为协程内部与外部世界的数据交换通道

3. 提供标准接口
    + 等待接口：
        - await_ready()：检查协程是否已完成
        - await_suspend()：挂起当前协程，将控制权转移给其他协程
        - await_resume()：获取协程的结果
    + 结果获取：
        - result()/get()：获取协程的计算结果
        - value()：获取生成器产生的当前值
    + 状态查询：
        - done()：检查协程是否已完成执行
        - is_ready()：检查结果是否可用
4. 异常处理机制
    - 捕获协程执行中抛出的异常
    - 提供异常传播机制
    - 支持通过promise_type自定义异常处理策略



## 五、协程帧（Coroutine Frame）：协程的“身体”

协程桢的概念是为了便于整体管理协程而提出来的抽象概念，并不能够被用户直接操作，但是理解它便于理解整个协程的运行原理。

可通过以下问题解释它的由来。

### 5.1 协程帧（Coroutine Frame）定义

+ 协程帧是 C++ 协程机制中由编译器自动分配的一块内存区域，用于存储协程运行所需的完整上下文状态。
+ 它在协程首次被调用时分配（通常在堆上），并在协程生命周期结束时被销毁。
+ 协程帧中包含以下内容：
    - promise_type 实例（最先构造）；
    - 函数参数（按值传递的会被拷贝进来）；
    - 局部变量（跨越 co_await / co_yield 的变量）；
    - 编译器生成的状态机数据（如当前挂起点索引、临时对象等）。

### 5.2 协程桢概念的必要性

+ 因为普通函数的执行依赖于调用栈，一旦函数返回，其栈帧就被销毁，所有局部状态丢失。

+ 但协程的核心特性是：

    - 可在执行中途暂停（co_await）；
    - 暂停后控制权返回调用者；
    - 之后可通过 resume() 从暂停点继续执行。
    - 这就要求：协程的执行状态必须跨越多次调用、跨越栈帧生命周期而依然有效。

+ 👉 因此，必须有一个独立于调用栈的内存区域来保存这些状态 —— 这就是 协程帧 的由来。

### 5.3 协程帧的内存分配与 promise_type 的存储要求

__协程帧必须分配在堆上（或等价内存）__

协程的生命周期不依赖调用栈，因为它可能在函数调用返回后很久才被 resume()。因此协程帧必须使用动态分配（默认在堆上），或通过自定义分配器管理，其生命周期由 coroutine_handle 控制，调用 destroy() 时才释放。这也是为什么协程有"堆开销"的原因 —— 但可通过无堆分配的优化（如 noexcept + 固定大小）避免。

__为什么 promise_type 对象必须作为协程帧的一部分？__

promise_type 不能放在栈上，也不能单独存在，原因如下：

1. 不能放在栈上：  
    如果 promise_type 在栈上（如作为局部变量），当协程函数首次返回时，其栈帧就会被销毁。但协程可能尚未执行完毕，后续 resume() 仍需访问 promise，此时访问已被销毁的栈内存会导致未定义行为。

2. 不能单独由用户管理：  
    用户无法预知协程何时开始、何时结束，也无法准确管理其生命周期（如多 resume、异常路径），且不同编译器生成的状态机差异大，难以统一接口。

因此，promise_type 必须作为协程帧的一部分，由编译器统一管理，在协程帧分配时构造，在 destroy() 时析构。这种设计既保证了协程的长期存活能力，又避免了手动管理生命周期的复杂性。

## 六、 协程操作句柄（std::coroutine_handle）：协程的“遥控器”

1. 什么是 coroutine_handle？

    + 一个轻量级、不透明的句柄，用于控制和操作协程；
    + 类似于“协程的指针”，指向协程帧（Coroutine Frame）；
    + 可复制、可移动，也可跨线程传递；
    + 是用户与协程之间进行交互的唯一控制接口。


2. 核心方法：

    | 方法 | 作用 |
    |------|------|
    | resume() | 恢复协程执行 |
    | destroy() | 销毁协程帧（释放内存） |
    | done() | 查询协程是否已完成 |
    | promise() | 访问 promise_type 实例 |
    | from_promise() | 从 promise 创建句柄 |
    | address() / from_address() | 底层地址操作（用于序列化） |

3. 使用示例：

    ```c++
    std::coroutine_handle<> handle = ...;
    if (!handle.done()) {
        handle.resume(); // 驱动协程
    }
    handle.destroy(); // 释放资源
    ```

    + 📌 coroutine_handle 是你与协程之间的“控制接口”，你可以有多个副本指向同一个协程帧。

## 七、awaiter 对象

### 7.1 什么是 awaiter 对象？

awaiter 对象是协程等待机制的核心组件，它定义了协程如何等待另一个协程或异步操作完成。当协程执行到 co_await 表达式时，会创建一个 awaiter 对象，该对象实现了协程挂起和恢复的具体逻辑。

### 7.2 awaiter 对象的接口要求

一个对象要成为 awaiter，必须满足以下接口要求：

1. awaitable 概念：
    + 必须可转换为 awaiter_type 类型（通常是其自身）
    + 该类型必须提供三个核心成员函数

2. 核心成员函数：
    ```c++
    /**
    * @brief 检查协程是否需要挂起等待
    * 
    * 此函数用于判断当前异步操作是否已经完成，决定协程是否需要立即恢复执行。
    * 
    * @return true 异步操作已完成，协程无需挂起，可直接继续执行
    * @return false 异步操作未完成，协程需要挂起等待
    * 
    * @note 当返回 true 时，编译器会优化掉挂起/恢复的开销，
    *       直接调用 await_resume() 实现零开销路径
     */
    bool await_ready() noexcept;

    /**
    * @brief 定义协程挂起时的操作和恢复策略
    * 
    * 此函数在协程遇到 co_await 表达式且 await_ready() 返回 false 时被调用，
    * 用于设置协程的挂起行为和恢复机制。
    * 
    * @tparam Promise 协程的 promise_type 类型
    * @param h 当前协程的句柄，包含协程执行地址和 promise 对象访问权限
    * @return coroutine_handle<> 返回目标协程句柄（对称协程模式）
    * @return void 返回 void（非对称协程模式）
    * @return bool 返回 false 表示立即恢复，true 表示继续挂起
    * 
    * @note 返回值决定协程的恢复方式：
    *       - 返回协程句柄：对称协程，当目标协程恢复时当前协程也会恢复
    *       - 返回 void：非对称协程，通过外部回调或事件恢复
    *       - 返回 bool：控制是否立即恢复
    * 
    * @example
    * struct my_awaiter {
    *     void await_suspend(coroutine_handle<> h) noexcept {
    *         // 注册异步回调，完成后恢复协程
    *         async_op([h]{ h.resume(); });
    *     }
    * };
    */
    template<typename Promise>
    auto await_suspend(coroutine_handle<Promise> h) noexcept;

    /**
    * @brief 获取协程恢复时的结果或执行恢复后的初始化
    * 
    * 此函数在协程从挂起状态恢复时被调用，用于获取异步操作的结果
    * 或执行恢复后的必要初始化工作。
    * 
    * @return decltype(auto) 异步操作的结果值或引用
    *         - 可以是值类型（int, string 等）
    *         - 可以是引用类型（避免拷贝）
    *         - 可以是 void（仅用于初始化）
    * 
    * @note 如果异步操作失败，应在此函数中抛出异常，
    *       异常会传播给 co_await 表达式的调用者
    * 
    * @example
    * struct my_awaiter {
    *     int await_resume() {
    *         return get_async_result();  // 返回异步操作结果
    *     }
    * 
    *     void await_resume() noexcept {
    *         setup_context();  // 仅用于初始化
    *     }
    * };
    */
    decltype(auto) await_resume();
    ```
3. 可选的异常处理接口：
    ```c++
    /**
    * @brief 无异常版本的恢复函数
    * 
    * 此函数是 await_resume() 的 noexcept 变体，用于在协程恢复时执行
    * 无异常抛出的操作。当协程作者确定恢复操作不会抛出异常时使用。
    * 
    * @note 使用场景：
    *       - 当异步操作结果保证有效，无需异常检查
    *       - 性能敏感的场景，避免异常处理开销
    *       - 仅用于执行恢复后的初始化，不涉及结果获取
    * 
    * @warning 如果此函数内部抛出异常，将导致程序终止（std::terminate）
    *          因此必须确保函数体不会抛出任何异常
    * 
    * @example
    * struct safe_awaiter {
    *     void await_resume() noexcept {
    *         // 仅执行无异常的初始化操作
    *         reset_state();
    *         update_context();
    *     }
    * };
    */
    void await_resume() noexcept;
    ```

### 7.3 与协程返回对象的关系

awaiter 对象与协程返回对象密切相关：

1. 协程返回对象作为 awaiter：

    - 协程返回对象本身通常也是一个 awaitable 类型
    - 当协程被其他协程 co_await 时，它就扮演了 awaiter 的角色
    - 通过实现 await_ready()/await_suspend()/await_resume() 接口定义自己的等待行为

2. 生命周期管理：
    - awaiter 对象的生命周期通常与 co_await 表达式的作用域绑定
    - 在 await_suspend() 中可以捕获当前协程的句柄，用于后续恢复

### 7.4 awaiter 的典型实现模式

1. 简单等待器：

```c++
struct simple_awaiter {
    bool await_ready() { return false; }
    void await_suspend(coroutine_handle<> h) {
        // 设置恢复回调或注册到事件循环
        h.resume();  // 立即恢复（仅用于演示）
    }
    void await_resume() {
        // 获取等待结果
    }
};
```

2. 异步操作适配器：

    + 将传统异步操作（如回调、Future）转换为协程可等待的形式
    + 在 await_suspend() 中注册回调，在回调中调用 h.resume()

3. 超时控制：

```c++
struct timeout_awaiter {
    std::chrono::milliseconds timeout;
    coroutine_handle<> h;
    
    bool await_ready() { return false; }
    void await_suspend(coroutine_handle<> ch) {
        h = ch;
        // 启动定时器，超时后恢复协程
        timer.expires_after(timeout);
        timer.async_wait([this](auto){ h.resume(); });
    }
    void await_resume() {
        // 检查是否超时
    }
};
```

### 7.5 awaiter 的设计要点

1. 无开销原则：
    + 当 await_ready() 返回 true 时，编译器会优化掉挂起/恢复的开销
    + 这种情况下相当于直接调用 await_resume()

2. 异常安全：
    + await_suspend() 中抛出的异常会导致当前协程恢复并抛出该异常
    + await_resume() 中的异常会被传播给等待者

3. 对称/非对称控制流：

    + 对称协程：await_suspend() 可以返回另一个协程句柄，实现协程链式调用
    + 非对称协程：await_suspend() 通常不返回（或返回 void），通过其他方式恢复

### 7.6 与 promise_type 的协作

awaiter 对象与 promise_type 协作完成协程的控制：

1. 定制等待行为：
    + promise_type 可以定义 await_transform() 方法
    + 在 co_await 表达式时，编译器会调用 await_transform() 对等待对象进行包装
    + 这允许协程作者定制不同的等待语义

2. 协程状态传递：

    + await_suspend() 接收当前协程的句柄，可以访问其 promise
    + 通过 h.promise() 可以获取或修改协程的内部状态

3. 功能划分： 
    在 C++ 协程中，功能应该放在 awaiter 还是 promise_type 中，主要取决于功能的作用范围和生命周期。与 co_await 表达式的作用域绑定，当 co_await 完成后通常会被销毁，短期存在，仅用于当前等待，适用于临时性的异步操作；与协程的生命周期绑定，从协程创建到 destroy() 调用期间一直存在，长期存在，管理协程的整体状态，适用于协程级别的逻辑。  
    + 放在 awaiter 中的功能
        - 与单个 co_await 表达式相关  
        功能仅影响当前 co_await 的行为，不涉及协程的整体控制。

        - 与异步操作的具体实现相关  
        例如：网络 IO、定时器、文件读取等，这些是 co_await 等待的具体对象。

        - 与协程的挂起/恢复逻辑相关  
        例如：决定是否挂起、如何恢复、获取异步结果等

    + 放在 promise_type 中的功能  
        - 与整个协程的生命周期相关  
        例如：协程的初始化、最终返回值、异常处理等。

        - 与协程的执行策略相关  
        例如：是否使用栈分配、协程调度策略等。

        - 与 co_await 的全局行为相关  
        例如：通过 await_transform() 修改所有 co_await 的行为。


### 7.7 实际应用示例

```c++
// 自定义的异步 IO awaiter
struct io_awaiter {
    tcp::socket& socket;
    boost::asio::mutable_buffer buffer;
    
    bool await_ready() const noexcept { return false; }
    
    template<typename Promise>
    void await_suspend(coroutine_handle<Promise> h) noexcept {
        auto& promise = h.promise();
        
        // 异步读取数据
        socket.async_read_some(buffer, 
            [h](boost::system::error_code ec, std::size_t bytes) {
                // 存储结果到 promise
                h.promise().result = {ec, bytes};
                h.resume();  // 通知协程继续执行
            });
    }
    
    std::pair<boost::system::error_code, std::size_t> await_resume() {
        return h.promise().result;
    }
};
```
```c++

// 在协程中使用
task<void> read_data(tcp::socket& socket) {
    std::array<char, 1024> buffer;
    auto [ec, bytes] = co_await io_awaiter{socket, buffer};
    if (!ec) {
        // 处理读取到的数据
    }
}
```

通过这种设计，awaiter 对象为协程提供了灵活的异步等待机制，使得协程能够自然地与各种异步系统和资源进行交互。


## 八、流程解析

整体上结构图如下所示。

```
+-----------------------+
| 协程返回对象           | ← 用户持有
| (task<T>, future)     |
+----------+------------+
           |
           | 持有 handle
           ▼
+----------------------------------+
|       协程帧 (堆上分配)           |
|                                  |
|  +----------------------------+  |
|  | promise_type 实例           | ← 控制逻辑
|  +----------------------------+  |
|                                  |
|  函数参数 (按值传递才安全)         |
|  局部变量 (跨越 suspend 点)       |
|  状态机数据 (如状态枚举、awaiter)  |
|                                  |
+----------------------------------+
```

让我们按照执行顺序，详细解析这个协程是如何工作的。

1. 步骤 1: 协程创建  
    ```c++
    task t = compute_value();
    ```
    + 发生了什么？
        i. 编译器为 compute_value 创建一个堆上的协程帧。
        ii. 在帧中构造 promise_type 实例。
        iii. 调用 promise_type::initial_suspend()。在我们的例子中，它返回 std::suspend_never，这意味着协程不会在初始时挂起，而是会立即开始执行函数体。
        iv. 调用 promise_type::get_return_object() 创建 task 对象 t，并将其返回给 main。
    + 输出：
    ```
    --- 创建协程 ---
    ```
2. 步骤 2: 协程开始执行  
    由于 initial_suspend 返回 std::suspend_never，协程创建后立即开始执行 compute_value 的函数体。
    ```
    task compute_value() {
        std::cout << "协程开始执行\n"; // <-- 立即执行
        suspend_later_awaiter awaiter;
        co_await awaiter;
        // ...
    }
    ```
    + 发生了什么？
        i. 输出 "协程开始执行"。
        ii. 创建 awaiter 对象，should_suspend 为 true。
        iii. 遇到第一个 co_await awaiter。
        iv. 调用 awaiter.await_ready()，返回 true。
        v. 因为 await_ready() 返回 true，co_await 表达式被认为是同步完成的，所以直接调用 await_resume()，然后继续执行。
    + 输出：
        ```
        协程开始执行
        awaiter: 第一次 await_ready() 返回 true，不挂起
        awaiter: 协程已恢复执行
        第一次 co_await 完成
        ```
3. 步骤 3: 遇到第二个 co_await  
    协程继续执行，遇到第二个 co_await awaiter。此时 awaiter 的 should_suspend 已被设为 false。
    ```
    co_await awaiter; // <-- 第二次
    ```
    + 发生了什么？
        i. 调用 awaiter.await_ready()，返回 false。
        ii. 因为 await_ready() 返回 false，co_await 表达式被认为是异步完成的。编译器会调用 await_suspend()，然后当前协程挂起。
        iii. iawait_suspend() 被调用，它输出 "协程被挂起..."，然后在一个新线程中启动一个定时器，1秒后调用 handle.resume() 来恢复这个协程。
    + 输出：
        ```
        awaiter: 第二次 await_ready() 返回 false，准备挂起
        awaiter: 协程被挂起，稍后恢复...
        ```
    此时，协程暂停执行，控制权回到 main 函数。

4. 步骤 4: 主线程等待与协程恢复  
    ```c++
    int main() {
        // ...
        std::cout << "\n--- 主线程继续工作，等待协程完成 ---\n";
        int result = t.result(); // <-- 阻塞直到协程完全结束
        // ...
    }
    ```
    + 发生了什么？
    i. main 线程输出信息后，调用 t.result()。
    ii. t.result() 内部调用 handle().resume()，这个调用会唤醒步骤3中挂起的协程，使其从 co_await awaiter; 的下一行继续执行。
    iii. 恢复时，必须调用 awaiter.await_resume()，输出 "协程已恢复执行"。
    iv. 协程继续执行，输出 "第二次 co_await 完成"。
    v. 遇到 co_return 42。调用 p.return_value(42)，然后调用 p.final_suspend()。
    vi. final_suspend() 返回 std::suspend_always，协程在 co_return 后再次挂起，等待被销毁。
    + 输出：
        ```
        --- 主线程继续工作，等待协程完成 ---
        主线程工作完成，等待协程结果...
        awaiter: 异步操作完成，准备恢复协程...
        awaiter: 协程已恢复执行
        第二次 co_await 完成
        ```

5. 步骤 5: 最终销毁
    ```
    int result = t.result(); // <-- t.result() 负责恢复并销毁
    ```

    + 发生了什么？
        i. t.result() 函数在 handle().resume() 中再次恢复协程，这次是为了执行 final_suspend() 之后的清理工作。
        ii. 协程函数体执行完毕，协程帧被标记为“已完成”。
        iii. t.result() 从 promise 对象中取出 value (42) 并返回。
        iv. main 函数接收结果并输出。
        v. main 函数结束，task 对象 t 析构，其析构函数调用 handle().destroy()，最终释放整个协程帧。
    + 输出：
    ```
    --- 最终结果: 42 ---
    ```

__要点分析：__

通过这个详细的示例，我们可以清晰地看到：

1. promise_type 是协程的“灵魂”：它定义了协程从生到死的宏观策略，决定着协程何时启动、何时结束以及如何返回结果。
2. awaiter 是异步操作的“手脚”：它封装了单个 co_await 的具体逻辑，决定着是否等待、如何等待以及如何恢复。它使得我们可以用统一、声明式的方式来处理各种不同的异步源。
3. co_await 是粘合剂：它将 awaiter 的行为无缝地集成到控制流中，使得异步代码的编写体验与同步代码别无二致。

理解 awaiter 和 promise_type 的职责划分与协作机制，是掌握 C++ 协程的关键。一旦你熟悉了它们的工作方式，你就能构建出高效、清晰且强大的异步应用程序。
